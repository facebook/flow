Objects

Object types describe the shape of JavaScript objects.

1const obj: {foo: number, bar: boolean} = {foo: 1, bar: true};
Try

Flow supports several variants:

  • Exact object types (default): exactly a set of properties, e.g. {a: number}.
  • Inexact object types: at least a set of properties, e.g. {a: number, ...}.
  • Indexed object types: used as a map, e.g. {[string]: boolean}.
TypeScript comparison

TypeScript's object types look identical but are open by default — {x: number} admits extra properties in TypeScript, while Flow's exact-by-default {x: number} rejects them. TypeScript's excess-property check only fires on direct literal assignment. The Flow spelling of TS's open shape is the inexact form {x: number, ...}. Two related divergences: Flow's class/interface/object subtyping is asymmetric because classes are nominal, and combining object types uses Flow's type-level spread {...A, b: T} rather than the TS intersection A & {b: T}.

When to use this​

Use object types for plain data and simple structures. When you need methods, inheritance, or nominal typing, use classes. When you need to accept both class instances and plain objects with the same shape, use interfaces.

Optional object type properties​

In JavaScript, accessing a property that doesn't exist evaluates to undefined. This is a common source of errors in JavaScript programs, so Flow turns these into type errors.

1const obj = {foo: "bar"};2obj.bar; // Error!prop-missingCannot get obj.bar because property bar is missing in object literal [1].
Try

If you have an object that sometimes does not have a property you can make it an optional property by adding a question mark ? after the property name in the object type.

1const obj: {foo?: boolean} = {};2
3obj.foo = true;    // Works!4obj.foo = 'hello'; // Error!incompatible-typeCannot assign 'hello' to obj.foo because string [1] is incompatible with boolean [2].
Try

In addition to their set value type, these optional properties can either be void or omitted altogether. However, they cannot be null.

1function acceptsObject(value: {foo?: string}) { /* ... */ }2
3acceptsObject({foo: "bar"});     // Works!4acceptsObject({foo: undefined}); // Works!5acceptsObject({});               // Works!6
7acceptsObject({foo: null});      // Error!incompatible-typeCannot call acceptsObject with object literal bound to value because in property foo: null [1] is incompatible with string [2].
Try

To make all properties in an object type optional, you can use the Partial utility type:

1type Obj = {2  foo: string,3};4
5type PartialObj = Partial<Obj>; // Same as `{foo?: string}`
Try

To make all properties in an object type required, you can use the Required utility type:

1type PartialObj = {2  foo?: string,3};4
5type Obj = Required<PartialObj>; // Same as `{foo: string}`
Try

Read-only object properties​

You can add variance annotations to your object properties.

To mark a property as read-only, you can use the readonly keyword.

1type Obj = {2  readonly foo: string,3};4
5function func(o: Obj) {6  const x: string = o.foo; // Works!7  o.foo = 'hi'; // Error!cannot-writeCannot assign 'hi' to o.foo because property foo is not writable.8}
Try

To make all object properties in an object type read-only, you can use the Readonly utility type:

1type Obj = {2  foo: string,3};4
5type ReadOnlyObj = Readonly<Obj>; // Same as `{readonly foo: string}`
Try

You can also mark your properties as write-only with the writeonly keyword.

1type Obj = {2  writeonly foo: string,3};4
5function func(o: Obj) {6  const x: string = o.foo; // Error!cannot-readCannot get o.foo because property foo is not readable.7  o.foo = 'hi'; // Works!8}
Try
Legacy sigil syntax

The readonly and writeonly keywords are enabled by default as of Flow 0.315. Older code may use the legacy + and - sigils (e.g. {+foo: string} instead of {readonly foo: string}, {-foo: string} instead of {writeonly foo: string}); these are still recognized but deprecated. See Modernizing Legacy Syntax for migration details.

Object methods​

Method syntax in objects has the same runtime behavior as a function property. These two objects are equivalent at runtime:

1const a = {2  foo: function () { return 3; }3};4const b = {5  foo() { return 3; }6}
Try

However, despite their equivalent runtime behavior, Flow checks them slightly differently. In particular, object properties written with method syntax are read-only; Flow will not allow you to write a new value to them.

1const b = {2  foo() { return 3; }3}4b.foo = () => { return 2; } // Error!cannot-writeCannot assign function to b.foo because property foo is not writable.
Try

Additionally, object methods do not allow the use of this in their bodies, in order to guarantee simple behavior for their this parameters. Prefer to reference the object by name instead of using this.

1const a = {2  x: 3,3  foo() { return this.x; } // Error!object-this-referenceCannot reference this from within method foo [1]. For safety, Flow restricts access to this inside object methods since these methods may be unbound and rebound. Consider replacing the reference to this with the name of the object, or rewriting the object as a class.4}5const b = {6  x: 3,7  foo(): number { return b.x; } // Works!8}
Try

Because method shorthand is read-only while a function-typed field is mutable, the two have different variance at the property. Method shorthand only triggers function-input contravariance, so a wider-input function can flow into a narrower-input method slot. A mutable function-typed field is invariant, so the same assignment is rejected:

1type ByMethod = {compare(a: string, b: string): number};2type ByField = {compare: (a: string, b: string) => number};3declare const wider: {compare: (a: string | number, b: string | number) => number};4
5const m: ByMethod = wider; // Works — method shorthand is read-only6const f: ByField = wider;  // Error — function-typed field is mutable and therefore invariantincompatible-typeCannot assign wider to f because in property compare: (a: string | number, b: string | number) => number [1] is not exactly the same as (a: string, b: string) => number [2].7
8type ByReadonlyField = {readonly compare: (a: string, b: string) => number};9const rof: ByReadonlyField = wider; // Works — readonly field is covariant, like method shorthand
Try

Marking the field readonly restores the covariant property position, which is why the third assignment succeeds where the mutable field rejected it. See Flow's method-parameter contravariance for the contrast with TypeScript's bivariance and the full mechanics.

Object type inference​

When you create an object value, its type is set at the creation point. You cannot add new properties, or modify the type of existing properties.

1const obj = {2  foo: 1,3  bar: true,4};5
6const n: number  = obj.foo; // Works!7const b: boolean = obj.bar; // Works!8
9obj.UNKNOWN; // Error - prop `UNKNOWN` is not in the object valueprop-missingCannot get obj.UNKNOWN because property UNKNOWN is missing in object literal [1].10obj.foo = true; // Error - `foo` is of type `number`incompatible-typeCannot assign true to obj.foo because boolean [1] is incompatible with number [2].
Try

If you supply a type annotation, you can add properties missing in the object value as optional properties:

1const obj: {2  foo?: number,3  bar: boolean,4} = {5  // `foo` is not set here6  bar: true,7};8
9const n: number | void = obj.foo; // Works!10const b: boolean = obj.bar; // Works!11
12if (b) {13  obj.foo = 3; // Works!14}
Try

You can also give a wider type for a particular property:

1const obj: {2  foo: number | string,3} = {4  foo: 1,5};6
7const foo: number | string = obj.foo; // Works!8obj.foo = "hi"; // Works!
Try

The empty object can be interpreted as a dictionary, if you supply the appropriate type annotation:

1const dict: {[string]: number} = {}; // Works!
Try

You may need to add type annotations to an object literal, if it references itself recursively (beyond simple cases):

1const Utils = { // Errorrecursive-definitionCannot compute a type for Utils because its definition includes references to itself [1]. Please add an annotation to these definitions [2] [3]2  foo() {3    return Utils.bar();4  },5  bar() {6    return 1;7  }8};9
10const FixedUtils = { // Works!11  foo(): number {12    return FixedUtils.bar();13  },14  bar(): number {15    return 1;16  }17};
Try

Exact and inexact object types​

Exact object types are the default.

Inexact objects (denoted with the ...) allow extra properties to be passed in:

1function method(obj: {foo: string, ...}) { /* ... */ }2
3method({foo: "test", bar: 42}); // Works!
Try

Note: This is because of "width subtyping".

But exact object types do not:

1function method(obj: {foo: string}) { /* ... */ }2
3method({foo: "test", bar: 42}); // Error!incompatible-typeCannot call method with object literal bound to obj because property bar is extra in object literal [1] but missing in object type [2]. Exact objects do not accept extra props.
Try

Intersections of exact object types are usually impossible types: a value would have to be exactly A and exactly B simultaneously, which is uninhabitable as soon as A and B differ at all. The operation that combines exact object types is object type spread, which mirrors the runtime semantics of value-level spread directly — own properties only, later keys overwrite earlier ones, and exactness propagates:

1type FooT = {foo: string};2type BarT = {bar: number};3
4type FooBarT = {...FooT, ...BarT};5const fooBar: FooBarT = {foo: '123', bar: 12}; // Works!6
7type FooBarFailT = FooT & BarT;8const fooBarFail: FooBarFailT = {foo: '123', bar: 12}; // Error!incompatible-typeCannot assign object literal to fooBarFail because property bar is extra in object literal [1] but missing in FooT [2]. Exact objects do not accept extra props.incompatible-typeCannot assign object literal to fooBarFail because property foo is extra in object literal [1] but missing in BarT [2]. Exact objects do not accept extra props.
Try

Object type spread​

Just like you can spread object values, you can also spread object types:

1type ObjA = {2  a: number,3  b: string,4};5
6const x: ObjA = {a: 1, b: "hi"};7
8type ObjB = {9  ...ObjA,10  c: boolean,11};12
13const y: ObjB = {a: 1, b: 'hi', c: true}; // Works!14const z: ObjB = {...x, c: true}; // Works!
Try

You have to be careful spreading inexact objects. Spreading inexact types only works when the spread appears before any named properties AND the resulting object is also inexact; otherwise make the spread source exact. The spread inexact object may have unknown properties that can override previous properties in unknown ways:

1type Inexact = {2  a: number,3  b: string,4  ...5};6
7type ObjB = { // Error!cannot-spread-inexactFlow cannot determine a type for object type [1]. Inexact [2] is inexact, so it may contain c with a type that conflicts with c's definition in object type [1]. Try making Inexact [2] exact.8  c: boolean,9  ...Inexact, // Errorincompatible-exactinexact Inexact [1] is incompatible with exact object type [2].10};11
12const x: ObjB = {a:1, b: 'hi', c: true};
Try

The same issue exists with objects with indexers, as they also have unknown keys:

1type Dict = {2  [string]: number,3};4
5type ObjB = { // Error!cannot-spread-indexerFlow cannot determine a type for object type [1]. Dict [2] cannot be spread because the indexer string [3] may overwrite properties with explicit keys in a way that Flow cannot track. Try spreading Dict [2] first or remove the indexer.6  c: boolean,7  ...Dict,8};9
10const x: ObjB = {a: 1, b: 2, c: true};
Try

Spreading an object value at runtime only spreads "own" properties, that is properties that are on the object directly, not the prototype chain. Object type spread works in the same way. Because of this, you can't spread interfaces, as they don't track whether a property is "own" or not:

1interface Iface {2  a: number;3  b: string;4}5
6type ObjB = { // Error!cannot-spread-interfaceFlow cannot determine a type for object type [1]. Iface [2] cannot be spread because interfaces do not track the own-ness of their properties. Try using an object type instead.7  c: boolean,8  ...Iface,9};10
11const x: ObjB = {a: 1, b: 'hi', c: true};
Try

Objects as maps​

JavaScript includes a Map class, but it is still very common to use objects as maps as well. In this use case, an object will likely have properties added to it and retrieved throughout its lifecycle. Furthermore, the property keys may not even be known statically, so writing out a type annotation would not be possible.

For objects like these, Flow provides a special kind of property, called an "indexer property." An indexer property allows reads and writes using any key that matches the indexer key type.

1const o: {[string]: number} = {};2o["foo"] = 0;3o["bar"] = 1;4const foo: number = o["foo"];
Try

An indexer can be optionally named, for documentation purposes:

1const obj: {[user_id: number]: string} = {};2obj[1] = "Julia";3obj[2] = "Camille";4obj[3] = "Justin";5obj[4] = "Mark";
Try

When an object type has an indexer property, property accesses are assumed to have the annotated type, even if the object does not have a value in that slot at runtime. It is the programmer's responsibility to ensure the access is safe, as with arrays.

1const obj: {[number]: string} = {};2obj[42].length; // No type error, but will throw at runtime
Try

Indexer properties can be mixed with named properties:

1const obj: {2  size: number,3  [id: number]: string4} = {5  size: 06};7
8function add(id: number, name: string) {9  obj[id] = name;10  obj.size++;11}
Try

You can mark an indexer property as read-only (or write-only) using variance annotations:

1type ReadOnly = {readonly [string]: number};2type WriteOnly = {writeonly [string]: number};
Try

Keys, values, and indexed access​

You can extract the keys of an object type using the keyof utility type:

1type Obj = {2  foo: string,3  bar: number,4};5
6type T = keyof Obj;7
8function acceptsKeys(k: T) { /* ... */ }9
10acceptsKeys('foo'); // Works!11acceptsKeys('bar'); // Works!12acceptsKeys('hi'); // Error!prop-missingCannot call acceptsKeys with 'hi' bound to k because property hi is missing in Obj [1].
Try

You can extract the values of an object type using the Values utility type:

1type Obj = {2  foo: string,3  bar: number,4};5
6type T = Values<Obj>;7
8function acceptsValues(v: T) { /* ... */ }9
10acceptsValues(2); // Works!11acceptsValues('hi'); // Works!12acceptsValues(true); // Error!incompatible-typeCannot call acceptsValues with true bound to v because: Either boolean [1] is incompatible with number [2]. Or boolean [1] is incompatible with string [3].
Try

You can get the type of an object type's specific property using indexed access types:

1type Obj = {2  foo: string,3  bar: number,4};5
6type T = Obj['foo'];7
8function acceptsStr(x: T) { /* ... */ }9
10acceptsStr('hi'); // Works!11acceptsStr(1); // Error!incompatible-typeCannot call acceptsStr with 1 bound to x because number [1] is incompatible with string [2].
Try

Arbitrary objects​

If you want to accept an arbitrary object safely, there are a couple of patterns you could use.

An empty inexact object {...} accepts any object:

1function func(obj: {...}) {2  // ...3}4
5func({}); // Works!6func({a: 1, b: "foo"}); // Works!
Try

It's often the right choice for a generic bounded to accept any object:

1function func<T extends {...}>(obj: T) {2  // ...3}4
5func({}); // Works!6func({a: 1, b: "foo"}); // Works!
Try

However, you can't access any properties off of {...}.

You can also try using a dictionary with unknown values, which would allow you to access any property (with a resulting unknown type):

1function func(obj: {readonly [string]: unknown}) {2  const x: unknown = obj['bar'];3}4
5func({}); // Works!6func({a: 1, b: "foo"}); // Works!
Try

The type Object is just an alias for any, and is unsafe. You can ban its use in your code with the unclear-type lint.

Common Issues​

Computed property access (invalid-computed-prop)​

Flow raises an invalid-computed-prop error when you use a computed key to access an object that doesn't have an indexer type. Flow needs to know the exact set of keys at the type level, so arbitrary computed access on regular object types is not allowed:

1const obj = {a: 1, b: 2};2
3const key: string = "a";4obj[key]; // Error! Can't use string to index into an object with known keys.invalid-computed-propCannot access object literal [1] with computed property using string [2].
Try

To fix this, you have a few options:

Use keyof typeof to narrow the key type when you know the key is valid:

1const obj = {a: 1, b: 2};2
3function getVal(key: keyof typeof obj): number {4  return obj[key]; // Works!5}
Try

Use an indexer type when the object is meant to be used as a dictionary:

1const obj: {[string]: number} = {a: 1, b: 2};2
3const key: string = "a";4obj[key]; // Works!
Try

Exponential type spread​

When you spread multiple conditional expressions into an object, Flow computes all possible combinations, which can grow exponentially. This causes an error when it exceeds Flow's limit:

1declare const cond1: boolean;2declare const cond2: boolean;3declare const cond3: boolean;4
5// Each conditional spread doubles the number of possible types6const obj = {7  ...cond1 ? {a: 1} : {},8  ...cond2 ? {b: 2} : {}, // Error: too many combinations to reason aboutexponential-spreadComputing object literal [1] may lead to an exponentially large number of cases to reason about because conditional [2] and conditional [3] are both unions, and spreading multiple unions produces one result for every combination of their members. Please use at most one union type per spread to simplify reasoning about the spread result. You may be able to get rid of a union by specifying a more general type that captures all of the branches of the union.9  ...cond3 ? {c: 3} : {},10  // Flow must consider 2^3 = 8 possible object types11};
Try

To fix this, annotate intermediate variables with a single type that covers all possibilities using optional properties:

1type Flags = {2  a?: number,3  b?: number,4  c?: number,5};6
7declare const cond1: boolean;8declare const cond2: boolean;9
10const flags1: Flags = cond1 ? {a: 1} : {};11const flags2: Flags = cond2 ? {b: 2} : {};12const obj: Flags = {...flags1, ...flags2};
Try
